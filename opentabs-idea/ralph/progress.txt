## Codebase Patterns
- Platform packages are in platform/, plugins in plugins/
- Each package has tsconfig.json (composite: true) and tsconfig.build.json (extends tsconfig.json)
- Root tsconfig.json uses project references for all platform/* packages, with `files: []` and `noEmit: true`
- Root scripts: `build` = `tsc --build`, `type-check` = `tsc --noEmit`
- bun-types must be installed as devDependency in packages that use Bun APIs (e.g., mcp-server)
- For `module.hot` (bun --hot), add a `bun-hot.d.ts` augmenting NodeModule interface since bun-types doesn't include it
- Package tsconfigs use `"module": "ES2022"`, `"moduleResolution": "bundler"`, `"target": "ES2022"`
- TypeScript is installed as root devDependency — workspaces share it
- Chrome extension uses @types/chrome as devDependency with `"types": ["chrome"]` in tsconfig
- Static HTML files (like offscreen.html) go in extension root (not src/) since tsc doesn't copy non-TS files
- Root `tsc --build` uses project references — it does NOT run each package's npm build script
- Chrome MV3 service workers support `"type": "module"` in manifest background config
- offscreen.html lives at `offscreen/offscreen.html` (extension root), references compiled JS at `../dist/offscreen/offscreen.js`
- Extension internal messaging uses `type` field: `ws:*` for WebSocket/offscreen, `bg:*` for background script operations
- Background↔offscreen use chrome.runtime.sendMessage; onMessage listener must return `true` for async response handling
- MCP server already handles ping/pong in websocket.message handler (US-001): method:"ping" → responds with method:"pong"
- Zod v3 is the peerDependency for plugin-sdk; installed as root devDependency for type resolution
- Plugin SDK exports: OpenTabsPlugin (abstract class), defineTool (identity function for generic inference), ToolError (extends Error with code field), ToolDefinition (interface)
- `opentabs build` CLI lives in plugin-sdk at src/cli.ts, exposed via bin field as `opentabs` command
- zod-to-json-schema is a dependency (not devDep) of plugin-sdk — needed at runtime by the CLI
- Bun.build with format:'iife' doesn't expose module exports; use a wrapper entry that imports + registers on globalThis
- IIFE adapter registration: `globalThis.__openTabs.adapters[pluginName] = plugin` (the extension reads from this)
- Plugin manifest field names use underscores: `url_patterns`, `input_schema`, `output_schema` (matching PRD spec)
- zodToJsonSchema options: `{ target: 'openApi3', $refStrategy: 'none' }` for clean JSON Schema without $ref
- MCP server uses low-level `Server` class (not `McpServer`) from @modelcontextprotocol/sdk to support raw JSON Schema tool registration
- MCP transport: `WebStandardStreamableHTTPServerTransport` (Web Standard Request/Response) at `/mcp` path — not Express/Node adapter
- Each MCP client session gets its own Server+Transport pair; all share the same state
- Extension WebSocket protocol: JSON-RPC 2.0 with methods `sync.full`, `tab.syncAll`, `tab.stateChanged`, `tool.dispatch`, `config.*`, `tool.invocation*`
- Server state is in `src/state.ts` (in-memory Maps for plugins, tabMapping, pendingDispatches); extension-protocol in `src/extension-protocol.ts`; MCP setup in `src/mcp-setup.ts`
- Config system: `src/config.ts` manages `~/.opentabs/config.json` with loadConfig/saveConfig; state.toolConfig is the in-memory mirror, persisted on changes via McpCallbacks.onToolConfigPersist
- Extension stores plugins in chrome.storage.local keyed by `plugin:{name}` — includes manifest fields + IIFE string
- IIFE injection uses chrome.scripting.executeScript with world:'MAIN' and indirect eval `(0, eval)(iifeCode)` to bypass CSP restrictions
- Extension handles sync.full (notification, no response), plugin.update (request with id → responds with reinjectedTabs), plugin.uninstall (request with id → responds with success)
- On extension startup (onInstalled, onStartup, top-level), stored plugins are re-injected into matching tabs
- Config default structure: `{ "plugins": [], "tools": {} }` — plugins = local filesystem paths, tools = prefixed name → boolean (absent = disabled)
- IIFE adapter registered at `globalThis.__openTabs.adapters[pluginName]` — has `isReady()`, `tools` array with `{name, handle(params)}`
- Tab state model: 'closed' (no matching tab), 'unavailable' (tab exists but isReady=false or adapter not found), 'ready' (tab exists and isReady=true)
- chrome.scripting.executeScript with world:'MAIN' and async func returns a Promise result — chrome resolves it before returning to the extension
- Tab state tracking: chrome.tabs.onUpdated (url/status changes) + chrome.tabs.onRemoved → re-compute state → send tab.stateChanged
- On WebSocket connect/reconnect: extension sends tab.syncAll with full state of all known plugins
- Plugin discovery: src/discovery.ts scans node_modules + local paths; plugins registered in state.plugins Map keyed by plugin name
- Plugin name derivation from npm: `opentabs-plugin-slack` → `slack`, `@scope/opentabs-plugin-name` → `scope-name`
- RegisteredPlugin.sourcePath stores the filesystem path for local plugins — used by file watcher to know which directories to watch
- File watcher (src/file-watcher.ts) uses node:fs watch with 200ms debounce; sendPluginUpdate() sends plugin.update to extension on IIFE change
- plugins/ directory is NOT a bun workspace member — each plugin has its own node_modules, bun.lock, and TypeScript
- Slack plugin lives at plugins/slack/ — scaffolded by create-opentabs-plugin, builds with `tsc && opentabs build`
- Local plugin paths in ~/.opentabs/config.json must be absolute paths for MCP server discovery
- Side panel is a React app in src/side-panel/ built with Bun.build (bundled) + @tailwindcss/cli (CSS); static HTML at side-panel/side-panel.html
- Root build script: `tsc --build && bun run --cwd platform/browser-extension build:side-panel` — tsc first, then Bun bundles React
- Background script message types for side panel: `sp:serverMessage` (server→panel), `sp:connectionState` (connection changes), `bg:send` (panel→server), `bg:getConnectionState` (query)
- Plugin sourcePath flows from MCP server (RegisteredPlugin.sourcePath) → sync.full/plugin.update → extension StoredPlugin.sourcePath — used for console logging link and file watcher
- RegisteredPlugin.npmPackageName stores original npm package name — used for version checking and update commands
- version-check.ts runs non-blocking on startup; results stored in state.outdatedPlugins and included in config.getState response
- Tailwind v4 uses `@import "tailwindcss"` in CSS and processes with `@tailwindcss/cli` — no config file needed
- React, ReactDOM, Tailwind, lucide-react are root devDependencies (shared via workspace)
